# 积分闯关与章节解锁设计

## 状态

- 日期：2026-07-12
- 状态：已确认（待用户审阅落盘稿）
- 范围：在现有 space 云同步与 XP 钱包之上，增加线性世界地图章节与专属奖励
- 目标用户：情侣双人 space；两人共享进度，日常打卡推进累计 XP
- 路线：世界地图里程碑（规则简单 + 地图/叙事包装）
- 配置策略：B 分阶段（阶段 1 种子章节可玩；阶段 2 管理端 CRUD）

## 背景

App 已具备双人 space、习惯打卡、XP 钱包（`balance` / `lifetimeEarned` / 流水）、奖励商城兑换与云端同步。用户希望增加「闯关」能力：**不扣积分**，只要累计获得 XP 达到门槛即可解锁章节，并获得章节专属奖励。

本功能是商城之外的第二条正向反馈线：商城消耗余额兑换货架奖励；闯关用累计获得量解锁叙事与徽章，二者并行、互不拖累。

## 一句话目标

为每个 space 提供一条共享的线性世界地图：`lifetimeEarned` 达标后自动解锁章节，用户手动领取徽章/称号与叙事内容；Tab「闯关」以进度舱 + 地图详情呈现游戏感，第一版用种子章节跑通，结构预留后续可配置与现实惊喜类型。

## 目标

1. 打卡提升 space 累计获得 XP，推进共享章节进度。
2. 章节严格按顺序解锁；达标后自动解锁，无需「付费入场」。
3. 解锁后手动领取专属奖励（徽章/称号 + 叙事），未领有明确提示。
4. 第 5 个 Tab 作为主入口；主页进度舱 + 详情世界地图，具备闯关氛围而非纯卡片列表。
5. 与奖励商城并行：兑换商城商品不降低闯关进度。
6. 阶段 1 预置 4–6 个种子章节即可玩；数据模型支持阶段 2 管理端配置。

## 非目标（阶段 1）

- 不扣 XP、不因断签/失败惩罚、不设战斗或排行榜。
- 不做章内多关卡、任务条件过关（仅累计 XP 门槛 + 线性顺序）。
- 章节专属奖励不进入商城 `rewards` 货架。
- 不做现实惊喜的完整兑现闭环（仅预留 `reward_type` 枚举值）。
- 不做完整章节 CMS / 管理端 CRUD（阶段 2）。
- 不改现有 XP 发放公式（沿用打卡/连击等规则）。

## 核心规则

| 项 | 约定 |
| --- | --- |
| 进度主体 | 每个 space 共享一条章节线 |
| 门槛指标 | `xp_wallet.lifetime_earned`（累计获得），与 `balance` 无关 |
| 解锁 | 严格线性：上一章已解锁 **且** `lifetimeEarned >= 本章 threshold` → 本章自动解锁 |
| 领奖 | 手动；`UNIQUE(space_id, chapter_id)` 幂等 |
| 撤销打卡 | `lifetimeEarned` 下降时 **不回锁** 已解锁章节；已领奖励不收回 |
| 与商城 | 并行；闯关不写 XP spend 流水 |

### 解锁算法（逻辑）

对已 `published` 且按 `sort_order` 排序的章节序列：

1. 第 1 章：若 `lifetimeEarned >= threshold`，则解锁。
2. 第 n 章：若第 n-1 章已解锁且 `lifetimeEarned >= 第 n 章 threshold`，则解锁。
3. 将 `adventure_progress.highest_unlocked_order` 更新为当前可达到的最大 `sort_order`（只增不减）。
4. 触发时机：读取闯关首页/地图时；以及 XP 交易成功导致 `lifetimeEarned` 上升后（幂等推进）。

## 用户流程

1. 用户完成打卡 → 现有 XP 服务写入 earn 流水 → `lifetimeEarned` 上升。
2. 系统幂等推进 `adventure_progress`（可能解锁新章节）。
3. 用户打开「闯关」Tab → 看到进度舱：当前章、累计 XP、距下一章差额、未领红点。
4. 进入世界地图 → 节点状态：已完成（已解锁且已领）/ 可领取（已解锁未领）/ 当前推进中 / 锁定。
5. 打开已解锁章节 → 阅读叙事 → 点击领取 → 写入 `adventure_claims`，徽章进入已获得集合。
6. 双方在同一 space 内看到一致进度与领取状态（同步/推送后刷新）。

## 界面结构

### Tab「闯关」主页（进度舱）

- 氛围头图或渐变背景（游戏感，非工具白底列表）。
- 大进度环/条：总章节进度或当前章内相对下一门槛的进度。
- 展示：当前章标题、space 累计 XP、下一章名称与还差 XP。
- 主 CTA：查看地图 / 领取奖励（有未领时强调）。
- Tab 角标：存在「已解锁未领取」时显示红点或数量。

### 世界地图详情

- 竖向（或可滚动）路径节点，强调场景与节点状态，避免「纯卡片堆叠」。
- 节点点击进入章节页或底部抽屉。
- 锁定节点可展示门槛与「需先完成上一章」。

### 章节页

- 标题、副标题、门槛 XP、叙事正文。
- 徽章预览（名称、描述、图标/占位）。
- 状态：锁定 / 可领取 / 已领取。
- 领取成功后轻量庆祝反馈。

### 徽章收藏

- 阶段 1 可先在闯关域内提供「已获徽章」入口或简易墙；不必单独 Tab。

### 管理入口（阶段 2）

- 复用隐藏 PIN 管理能力，增加章节配置：门槛、文案、徽章、上下架、排序。

## 数据模型

所有业务数据按 `space_id` 隔离（章节配置可采用「全局模板 + space 实例」或「注册 space 时复制种子」；阶段 1 推荐 **注册/迁移时复制种子到 space**，便于阶段 2 每 space 定制）。

### adventure_chapters

章节配置与展示内容。

| 字段 | 说明 |
| --- | --- |
| id | 主键 |
| space_id | 所属 space |
| sort_order | 线性顺序，从 1 递增 |
| title / subtitle | 展示文案 |
| story_md 或 story_text | 叙事正文 |
| threshold_lifetime_xp | 解锁所需累计获得 XP |
| badge_name / badge_description | 徽章文案 |
| badge_icon_key 或 badge_emoji | 图标；阶段 1 可用 emoji 占位 |
| reward_type | `badge_story`（阶段 1）/ `real_pending`（预留） |
| map_theme_key / node_style | 地图节点视觉键（可选） |
| status | `published` / `draft` / `archived` |
| created_at / updated_at | 时间戳 |

约束建议：

- `UNIQUE(space_id, sort_order)` 在 published 集合内保持顺序清晰。
- 仅 `published` 参与解锁计算。

### adventure_progress

每个 space 一条进度。

| 字段 | 说明 |
| --- | --- |
| space_id | PK |
| highest_unlocked_order | 当前已解锁的最大 sort_order，初始 0 |
| updated_at | 更新时间 |

说明：具体「是否可领」由 `highest_unlocked_order` + `adventure_claims` + 章节配置共同推导，不必为每章存一行进度。

### adventure_claims

领奖记录。

| 字段 | 说明 |
| --- | --- |
| id | 主键 |
| space_id | space |
| chapter_id | 章节 |
| claimed_at | 领取时间 |
| claimed_by | 领取账号 id（可空但建议记录） |

约束：`UNIQUE(space_id, chapter_id)` 保证幂等。

### 与现有表的关系

- **只读** `xp_wallet.lifetime_earned` 作为门槛源。
- **不** 为闯关写入 `xp_transactions` spend。
- **不** 复用 `rewards` / `reward_redemptions` 作为章节货架；专属奖励走 `adventure_*`。
- 同步：新资源纳入现有 space 数据同步与 WebSocket 失效刷新机制。

## API / 服务职责（阶段 1）

1. **列表进度**：返回章节列表（含解锁/领取状态）、progress、当前 `lifetimeEarned`、下一章差额。
2. **推进解锁**：幂等函数 `advanceAdventureProgress(spaceId)`，在 XP 上升与读进度时调用。
3. **领取**：`claimChapter(spaceId, chapterId, accountId)`  
   - 校验已解锁  
   - 插入 claim（冲突则返回已领取）  
   - 推送同步失效  
4. **种子/迁移**：新 space 写入默认章节；已有 space 补种并按当前累计一次性 `advance`。

权限：

- 读进度 / 领奖：space member。
- 改章节配置：owner/admin（阶段 2）。

## 种子内容（产品需准备）

阶段 1 建议 4–6 章。实现前需确定：

1. 世界线名称与 Tab 中文名（如「旅程」「闯关」「地图」）。
2. 每章：`title`、`subtitle`、叙事正文、徽章名与描述。
3. 门槛表示例（可调）：50 / 150 / 300 / 500 / 800 / 1200。
4. 视觉占位策略：emoji 徽章 + 主题色节点；插画可后补。

未在设计阶段锁定具体文案时，工程可用可替换占位文案落地，上线前替换。

## 分期交付

### 阶段 1（本设计范围）

- 三表 + 迁移 + 种子
- 推进解锁 + 领取 + 同步
- 第 5 Tab、进度舱、地图详情、章节页、红点
- 单元/集成测试与双人一致性验证
- 徽章简易收藏入口

### 阶段 2

- 管理端章节 CRUD、排序、上下架
- 徽章/地图插画上传（可复用 R2）
- 现实惊喜类型的待兑现/已兑现流程（若启用）

### 可选增强

- 更强过关动画、地图 parallax
- 独立徽章墙页
- 章节主题皮肤与 App 主题联动

## 验收标准（阶段 1）

1. 当 `lifetimeEarned` 达到第 n 章门槛且第 n-1 章已解锁时，打开闯关可见第 n 章已解锁，无需手动「解锁」操作。
2. 未达门槛或上一章未解锁时，章节保持锁定。
3. 领取成功后徽章/状态可见；重复领取不产生重复记录。
4. 在商城兑换导致 `balance` 下降时，章节进度不变。
5. 撤销打卡导致累计下降时，已解锁章节不回锁，已领不收回。
6. 同一 space 两名成员在同步后看到一致的解锁与领取状态。
7. UI 具备进度舱 + 路径地图的闯关表达，而非纯列表卡片墙。

## 风险与开放项

| 项 | 说明 | 建议 |
| --- | --- | --- |
| Tab 拥挤 | 5 个 Tab 在小屏偏满 | 阶段 1 接受；可后续做图标/文案精简 |
| 内容质量 | 叙事与徽章决定「游戏感」 | 上线前替换占位文案；视觉可迭代 |
| 配置复制 vs 全局模板 | 影响阶段 2 运营方式 | 阶段 1 用 per-space 种子复制 |
| 门槛数值 | 过低太快打完，过高长期看不到进度 | 用可调种子；观察 1–2 周再改 |
| 章节名/世界观 | 尚未在 brainstorm 中锁定 | 实现计划前由产品给定或用占位 |

## 设计决策摘要

- 玩法骨架：世界地图里程碑，不是章内多关卡。
- 经济关系：门槛看累计获得，不扣余额；与商城并行。
- 社交关系：space 共享进度与章节线。
- 交互：自动解锁 + 手动领奖。
- 奖励：徽章/称号 + 叙事；预留现实惊喜。
- 入口：第 5 Tab。
- 配置：阶段 1 种子，阶段 2 管理端。
