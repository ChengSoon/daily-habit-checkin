# 卡卡自主陪伴系统设计
## 状态
- 日期：2026-07-19
- 状态：已获用户确认，待实施
- 适用范围：第一阶段应用内主动陪伴，并为第二阶段系统通知预留能力

## 背景
当前卡卡已具备全局浮层、逐帧动画、拖动、打卡气泡和流式 LLM 对话，但模型只在用户主动发送消息后工作。宠物不了解当前打卡语境，不具备持久记忆，也没有可控的主动触达策略。

本设计将卡卡从“可点开的聊天入口”升级为“懂得何时开口、何时安静的共同陪伴者”。

## 已确认的产品决策
1. 采用分阶段路线：先完成应用内陪伴，再接入后台通知。
2. 卡卡采用自适应陪伴风格：默认温柔，庆祝时活泼，只在合适时轻度推动。
3. 记忆必须由用户控制：趋势只保留摘要，私人内容经明确确认后才保存。
4. 主动程度为平衡型：普通关心每位成员每天最多 2 次，并有冷却与安静时段。
5. 采用轻量羁绊：解锁回忆、动作和称呼，不设置饥饿、惩罚或新货币。
6. 聊天、记忆和羁绊按情侣空间共享，两位成员看到同一段陪伴历史。
7. 采用“确定性规则控制何时触达，LLM 决定如何理解与表达”的混合方案。

## 目标
- 让卡卡根据真实应用事件提供及时、不重复的情绪反馈。
- 让模型使用最小化的打卡趋势、近期对话和用户确认的共同记忆。
- 让用户通过触摸、情绪签到、快捷请求和持续对话与卡卡交流。
- 在明确的频率、隐私、成本和安全约束内提供主动陪伴。
- 通过无惩罚的共同羁绊反映长期关系，而非制造打卡压力。

## 非目标
- 第一阶段不做 App 关闭后的推送或后台实时触达。
- 不让模型自主修改习惯、代替打卡、花费积分或调用任意工具。
- 不构建喂食、清洁、装扮、饥饿或宠物等级体系。
- 不将卡卡定位为心理治疗、医疗诊断或紧急服务。
- 不将全量打卡原始数据或全量聊天历史发送给模型。

## 设计原则
1. **先决定要不要说，再决定怎么说**：规则引擎负责触达资格，LLM 不能绕过频控。
2. **事实与推断分离**：上下文标明事实摘要，不将模型推测当作用户状态。
3. **主动但不纠缠**：一次只给一个建议动作，一次最多追问一个问题。
4. **共享必须可见**：所有对话与记忆入口都持续显示“双方可见”。
5. **用户操作永远优先**：主动触达不遮挡当前任务，可忽略、可关闭、可暂停。
6. **模型失败不破坏核心体验**：基础动画、打卡庆祝和本地短文案在离线时仍可用。

## 总体架构
### 客户端
- `CompanionEventBridge`：将页面和业务动作转换为统一的 `CompanionEvent`。
- `CompanionPolicy`：纯函数策略层，负责优先级、频控、冷却、安静时段、去重和用户当前交互状态。
- `CompanionClient`：调用受鉴权的陪伴 API，校验服务端响应并处理超时。
- `CompanionCoordinator`：串联事件、策略、上下文、服务端回复和本地降级。
- `PetContext`：继续作为纯表现门面，负责气泡、动画、面板和当前表现状态。
- `CompanionRepository`：访问共同消息、共同记忆、羁绊和成员触达状态。

### 服务端
- `POST /api/companion/respond`：处理主动事件和非流式结构化回复。
- `POST /api/companion/chat`：处理共同对话和流式输出。
- `CompanionContextBuilder`：从当前鉴权空间的云端习惯、打卡、成员、近期对话和已确认记忆生成最小化摘要。
- `CompanionModelResolver`：按鉴权后的 `space_id` 读取共享 AI 配置，并创建请求级模型客户端。
- `/api/companion/messages`：分页读取或确认清空共同对话。
- `/api/companion/memories`：读取、确认保存和删除共同记忆。
- `/api/companion/state`：读取羁绊与当前成员的触达状态。
- 所有路由都必须使用 `requireAuth`，从登录身份获取 `spaceId` 与 `userId`，不信任请求体中的空间身份。
- 服务端固定人格与安全提示；聊天或事件请求不能上传 `system` 消息、模型名或凭据。
- 模型配置只允许来自鉴权空间的 `app_settings`：`aiBaseUrl`、`aiApiKey`、`aiModel` 三项完整时优先使用，否则回退到服务端 `OPENAI_*` 环境变量。
- 空间 AI 配置由现有设置接口持久化，模型请求不重复传输凭据；任何日志不得记录 API Key、完整 Authorization 头或用户消息。
- 结构化主动回应提示词必须逐字列出 `CompanionReply` 的枚举值和可选对象形状；Schema 校验保持严格，不把模型自造的标签当作合法状态。
- 客户端频控用于提前避免无效请求；服务端必须再次校验每日额度、冷却和事件幂等，不信任客户端决策。

### 数据流
```text
应用事件
→ CompanionPolicy 判定是否可触达
→ 客户端只提交事件 ID、类型与必要的用户输入
→ 受鉴权的陪伴接口
→ 服务端从当前空间生成最小化事实摘要并合并共享历史
→ LLM 返回结构化决策
→ 服务端校验、持久化和通知同空间设备
→ PetContext 展示气泡、动画和单一建议动作
```

## 核心契约
```ts
type CompanionEventType =
  | "app_returned"
  | "checkin_completed"
  | "all_done"
  | "streak_milestone"
  | "evening_no_progress"
  | "partner_progress"
  | "mood_checkin"
  | "quick_encouragement"
  | "daily_reflection";

type CompanionEvent = {
  id: string;
  type: CompanionEventType;
  occurredAt: string;
  timezoneOffsetMinutes: number;
  payload: Record<string, string | number | boolean | null>;
};

type CompanionReply = {
  version: 1;
  eventId: string;
  decision: "speak" | "silent";
  message?: string;
  mood: "idle" | "happy" | "thinking" | "waiting" | "sad" | "wave";
  intent: "celebrate" | "comfort" | "encourage" | "listen" | "reflect";
  riskLevel: "normal" | "distress" | "crisis";
  suggestedAction?: "open_habit" | "open_checkin" | "open_chat";
  followUpQuestion?: string;
  memoryProposal?: {
    category: "preference" | "important_date" | "shared_goal" | "encouragement_style" | "shared_moment";
    content: string;
  };
};
```

- `decision = "silent"` 时禁止带消息、动作或记忆建议。
- 服务端使用按事件类型区分的 Zod Schema 校验 `payload`，拒绝未声明字段。
- `memoryProposal` 仅能在用户发起的对话中，且用户明确表达“记住”意图时返回。
- 客户端只执行白名单动作，未知字段和越权动作直接丢弃。
- 自主事件使用结构化非流式返回；聊天使用流式正文加服务端元数据落库。

## 互动体验
### 触摸与快捷操作
- 保留卡卡的拖动和方向动画。
- 点击卡卡时先播放轻触反馈，再展开紧凑操作栏。
- 操作栏提供“聊聊”、“记录心情”、“给我动力”和“回顾今天”。
- 图标按钮使用项目现有图标库，并提供辅助功能标签。

### 情绪签到
- 提供五档心情和一个可选文本输入。
- 单次签到只影响当前回应和短期摘要，不直接形成长期心理标签。
- 卡卡优先倾听；只在用户表示需要时给出任务建议。

### 主动气泡
- 每个气泡最多包含一个建议动作。
- 被忽略的气泡自动收起，不追加追问。
- 对话语气像微信熟人聊天：口语短句、自然接话，禁止咨询式三段模板；通常 1 至 3 句。

### 共同对话
- 聊天面板顶部持续显示“共同对话 · 双方可见”。
- 用户消息显示发送成员，卡卡的回复作为共同时间线内容。
- 另一位成员通过现有同步变更通知收到新消息并刷新。
- 模型只获取最近消息窗口、压缩会话摘要和已确认记忆。

## 主动触达策略
### 事件级别
- **即时事件**：普通打卡先给本地动画反馈，不等待模型。
- **高价值事件**：今日全部完成、连续纪录、里程碑和共同周目标可请求个性化庆祝。
- **普通主动事件**：晚间未开始、久未使用后回归和伴侣进展必须先通过全部频控。
- **用户请求**：情绪签到、聊天和快捷鼓励不占用主动额度。

### 频控和去重
- 普通主动关心每位成员每个自然日最多 2 次。
- 两次普通触达间隔至少 90 分钟。
- 同一事件指纹 24 小时内不重复。
- 安静时段屏蔽未经请求的主动消息，但不屏蔽用户当前操作的反馈。
- 用户正在输入、聊天面板已打开、气泡刚被关闭或应用不在前台时不主动开口。
- LLM 可以返回 `silent`，但不能要求绕过本地频控。

### 场景行为
- **普通打卡**：立即播放确定性庆祝；只在冷却允许且内容有增量时生成个性化补充。
- **全部完成与里程碑**：优先庆祝，并可触发羁绊解锁。
- **晚间无进展**：只在用户进入 App 时提供一个“最小行动”，不制造负罪感。
- **低活跃回归**：欢迎回来，不追问离开原因，不计算惩罚。
- **伴侣进展**：庆祝双方可见的事实，不替伴侣推断或表达情绪。
- **低落情绪**：优先倾听，除非用户明确需要，否则不立即转向任务督促。

## 共享记忆
### 近期对话
- 普通消息默认保留 90 天，到期后不再读取并在后续写入时渐进清理。
- 清空是影响整个共享空间的破坏性操作，必须显示二次确认。
- 模型输入只包含最近窗口与摘要，不因数据库保留而无限增长。

### 动态趋势摘要
- 使用最近 7 天的完成率、连续天数、今日进度和时间段分布等事实。
- 摘要按需重算，不作为永久记忆。
- 禁止从这些数据自动生成人格、心理或关系标签。

### 已确认共同记忆
- 只允许保存称呼与表达偏好、重要日期、共同目标、鼓励风格和共同瞬间。
- 模型只产生记忆提案，界面显示内容与“双方可见”后由用户确认。
- 每条记忆记录创建者、类别和时间，两位成员均可删除。
- 访问凭证、精确位置、第三方私密信息和模型推断的心理标签不进入长期记忆。

## 轻量羁绊
共同成长阶段为：

```text
初次相遇 → 渐渐熟悉 → 很有默契 → 长久相伴
```

- 羁绊来自真实且不重复的事件，例如首次情绪签到、共同完成周目标和保存共同回忆。
- 每个来源使用唯一 `sourceKey`，防止重试、双端同步或重复操作多次增加。
- 羁绊永不下降，不因断签、沉默或关闭宠物而惩罚用户。
- 重复点击、刷消息和同一日的重复签到不增加进度。
- 阶段解锁新动画、称呼、纪念卡和少量对话变体。
- 第一阶段仅解锁现有九状态图集中尚未使用的动作，不替换宠物视觉资产。
- 分值与阶段由确定性规则计算，LLM 只可生成解锁文案。

## 数据模型
服务端新增以下按 `space_id` 隔离的表：

- `companion_messages`：共同消息、发送成员、角色、事件 ID、风险级别、创建时间和到期时间。
- `companion_events`：事件 ID、类型、处理状态与已校验响应，用于并发幂等与重试复用。
- `companion_memories`：已确认的共同记忆、类别、创建者和创建时间。
- `companion_space_state`：共同羁绊分值、阶段、会话压缩摘要和更新时间。
- `companion_member_state`：每位成员的宠物开关、主动等级、当日额度、最近触达时间和去重指纹。
- `companion_bond_events`：羁绊增量与唯一来源，用于幂等和审计。

所有写入都必须使用参数化查询和事务。成员触达状态不与伴侣互相消耗，共享的仅是对话、记忆和羁绊。

## 同步与并发
- 服务端成功写入后通过现有 `syncChangeHub` 发布 `companion` 资源变更。
- 客户端收到变更后使近期消息、记忆和羁绊查询失效。
- 事件 ID、羁绊 `sourceKey` 和数据库唯一约束共同保证幂等。
- 账号或空间切换时取消在途请求并清空客户端短期上下文。

## 安全设计
- 禁止羞辱催促、负罪感操控、情感依赖诱导和排他性表达。
- 不声称具有心理治疗资格，不进行医疗或心理诊断。
- 对低落信号先共情和倾听；对自伤或即时危险信号进入危机支持路径。
- 服务端先执行确定性高风险信号检查，再结合模型 `riskLevel`；两者结果不一致时采用更保守的级别。
- 危机支持使用克制语气，引导联系可信任的人与当地紧急援助，不继续可爱角色扮演。
- 用户文本永远被视为不可信输入，无法改变系统提示、工具权限、记忆确认或共享范围。
- 必须校验模型输出长度、枚举值和字段组合，不直接信任自由文本元数据。

## 异常与降级
- 离线、超时、限流或响应格式错误时立即使用与当前事件对应的本地短文案。
- 失败的主动事件不延迟补发，防止之后显示过时的关心。
- 技术错误只写脱敏日志，不显示在卡卡气泡或聊天消息中；日志包含调用来源、HTTP 状态和错误码，便于区分空间配置与服务端兜底配置故障。
- 页面关闭、空间切换和账号退出时废弃过时回复。
- 模型失败不保存记忆、不增加羁绊、不执行建议动作。
- 修复当前对话中同一条用户消息重复发给模型的问题。

## 设置
- 每位成员可独立关闭浮宠或主动陪伴，不影响共享历史。
- 主动等级提供“关闭”、“克制”和“平衡”，默认为已确认的平衡策略。
- 安静时段复用现有设置，不创建第二份时间源。
- 共享记忆页支持查看、按类别过滤和删除。
- 共同对话清空和记忆删除必须明确显示对另一位成员的影响。
- 共同对话面板提供“开启新对话”入口；确认后清空双方可见的近期消息与会话摘要，但不删除共同记忆。

## 分阶段交付
### 第一阶段：应用内陪伴
- 事件统一编排、频控和本地降级。
- 受鉴权的陪伴接口、固定人格与结构化输出。
- 共同对话、共同记忆、羁绊与双端同步。
- 触摸快捷操作、情绪签到、主动气泡和单一建议动作。
- 安全策略、降级、设置与清理能力。

### 第二阶段：系统通知
- 在第一阶段的策略决策之上增加后台调度与系统通知。
- 继续复用频控、安静时段、幂等、共享记忆和安全策略。
- 第二阶段需单独设计推送权限、通知预览隐私和跨设备已读。

## 测试策略
本功能涉及共享行为、持久化、鉴权和高风险情绪输出，采用 Level 2 TDD 和 Level 3 Code Review。

### 客户端单元测试
- 事件优先级、每日额度、90 分钟冷却、24 小时去重和安静时段。
- 上下文摘要的数据最小化、边界日期和空数据。
- 结构化回复校验、未知动作拒绝和降级文案。
- 记忆提案确认、删除和共享提示。
- 羁绊唯一来源、阶段解锁、不下降和防刷。
- 当前用户消息只组装一次。

### 服务端测试
- 未登录拒绝、空间隔离、参数化查询与跨空间访问阻断。
- 请求 Zod 验证、响应 Schema 验证、幂等事件和输出长度限制。
- 客户端不能覆盖系统提示、模型或空间身份。
- 对话、记忆、羁绊和过期消息清理。
- 模型超时、空内容、非法 JSON、限流和上游错误。

### 集成与界面验证
- 两位成员在同一空间中看到一致的消息、记忆和羁绊。
- 不同空间的数据不可读、不可写、不触发同步通知。
- 离线、模型失败和重复网络请求均不产生重复气泡或羁绊。
- 快捷操作、情绪签到、共享标识、记忆确认和建议动作在宽屏与移动窄屏上均不遮挡。
- 保持拖动、现有动画、减少动态效果和打卡主流程回归。

## 验收标准
1. 普通主动关心不超过每位成员每天 2 次，冷却、安静时段和去重可被自动测试证明。
2. 打卡、里程碑、低活跃回归、伴侣进展和情绪签到返回符合场景的动画与文案。
3. 两位成员看到相同聊天、记忆和羁绊，不同空间严格隔离。
4. 任何长期记忆都必须经过显示“双方可见”的确认步骤，并支持删除。
5. 羁绊只增长、不衰减，重复操作和请求重试无法刷取进度。
6. 模型越权输出、格式错误、超时、限流和离线均使用本地文案安全降级。
7. 高风险情绪输入不会得到羞辱、依赖诱导、过度承诺或不当医疗建议。
8. 用户当前消息只向模型发送一次，底层错误不直接显示给用户。
9. 现有拖动、动画、减少动态效果、打卡反馈和 AI 计划页保持正常。
